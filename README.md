# tier4-qa-task

[practice-software-testing](https://github.com/testsmith-io/practice-software-testing)（Toolshop）を対象に、手動テストケースの設計と Playwright + TypeScript による自動化を行うプロジェクト。テスト対象は公開デモサイトではなく、`test-target` サブモジュールとしてローカル Docker で起動したものを使う。公開デモは世界中の利用者が同じアカウントを共有しており、他者のログイン失敗によるアカウントロックでこちらのテストまで巻き添えに落ちるため。

## セットアップと実行

前提: Node 24、pnpm 10.28.2（`package.json` の `packageManager` フィールドを corepack が読む）、Docker（`docker compose` が使えること）。

```bash
# 1. 依存関係をインストール
pnpm install

# 2. 環境変数ファイルを用意する
cp .env.example .env

# 3. テスト対象アプリをローカル Docker で起動する
cd test-target
docker compose up -d   # 初回はイメージビルドが入るため数分かかる
cd ..

# 4. テストを実行する
pnpm test
```

`pnpm test` は `dotenvx run -- playwright test` を実行し、`.env` の値を読み込んだ上で `playwright.config.ts` の `globalSetup`（`config/global-setup.ts`）が DB を初期シード状態へ戻してからテストを開始する。

`.env` が持つ2つの変数の意味:

| 変数       | 既定値                  | 用途                                                                                                                       |
| ---------- | ----------------------- | -------------------------------------------------------------------------------------------------------------------------- |
| `BASE_URL` | `http://localhost:4200` | Playwright がブラウザで開く UI（Angular）のアドレス。`playwright.config.ts` の `use.baseURL`                               |
| `API_URL`  | `http://localhost:8091` | テストコードが `request` フィクスチャで直接叩く REST API のアドレス。テスト用アカウントの登録（`config/account.ts`）に使う |

## 手動テストケース設計

### TC-001: ログイン（9 シナリオ）

対象: `tests/login.spec.ts`。各シナリオは、そのテスト専用に API 登録したアカウント（`config/account.ts` の `registerAccount`）のメール/パスワードを土台に、入力値を変えて実行する。

| #   | シナリオ名                                | メール入力             | パスワード入力             | 期待結果                                                                                            | 種別   |
| --- | ----------------------------------------- | ---------------------- | -------------------------- | --------------------------------------------------------------------------------------------------- | ------ |
| 1   | 正常系: 正しい認証情報でログインできる    | 登録アカウントのメール | 登録アカウントのパスワード | `/account` に遷移し、見出し "My account" が表示される                                               | 正常   |
| 2   | 誤ったパスワード                          | 登録アカウントのメール | `wrong-password`           | ログインエラー "Invalid email or password" が表示される                                             | 異常   |
| 3   | パスワード空欄                            | 登録アカウントのメール | （空文字）                 | パスワードエラー "Password is required" が表示される                                                | 異常   |
| 4   | パスワード境界値: 2文字（無効）           | 登録アカウントのメール | `ab`                       | パスワードエラー "Password length is invalid" が表示される                                          | 境界値 |
| 5   | パスワード境界値: 3文字（長さ検証は通過） | 登録アカウントのメール | `abc`                      | パスワード長エラーは表示されず、かつログインエラー "Invalid email or password" が表示される（後述） | 境界値 |
| 6   | メール空欄                                | （空文字）             | 登録アカウントのパスワード | メールエラー "Email is required" が表示される                                                       | 異常   |
| 7   | メール形式不正: 1文字                     | `a`                    | 登録アカウントのパスワード | メールエラー "Email format is invalid" が表示される                                                 | 異常   |
| 8   | メール形式不正: @と特殊文字               | `a@@invalid!!.com`     | 登録アカウントのパスワード | メールエラー "Email format is invalid" が表示される                                                 | 異常   |
| 9   | メール境界値: @+1文字（形式検証は通過）   | `a@b`                  | 登録アカウントのパスワード | メール形式エラーは表示されず、かつログインエラー "Invalid email or password" が表示される（後述）   | 境界値 |

**境界値の根拠（実測）**: パスワードの長さ検証は 2 文字を無効、3 文字を通過として扱う（シナリオ #4 / #5）。メール形式検証は `a@b`（`@` の前後に 1 文字ずつ）を通過として扱う（シナリオ #9）。これらは仕様書ではなく、ローカルの `test-target` に実際に入力して応答を観測して確定した値であり、アプリのバリデーションルールが変われば境界も変わる。

**#5・#9 でエラー非表示に加えて認証エラー表示まで確認する理由**: 長さ/形式のエラーが表示されないことだけを見た検証では、①入力がサーバまで届きバリデーションを通過した結果としてエラーが出ない場合と、②JS の不具合やイベントハンドラの不備でボタンが反応せず何も送信されていない場合を区別できない。後者が起きていてもテストは「エラーが無い」ことしか見ていないため誤って合格にしてしまう。そこで、対応するエラーが非表示であることに加えて、認証エラー "Invalid email or password" が表示されることまで確認する。認証エラーが出るのは、サーバがその入力を受理しログイン処理（認証情報の照合）まで進んだことを示す正の証拠になる。

### TC-002: 商品購入ジャーニー（1本）

対象: `tests/purchase.spec.ts`。ゲスト状態から商品を選び、チェックアウト内サインインを経て注文を確定するまでの一連の流れを 1 本のシナリオで検証する。

| ステップ                                          | 手順                                                                                                                                               | 期待結果                                                                                                                                                                                                                                |
| ------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Arrange: ホームから商品詳細へ遷移する             | ホームを開き、固定の商品名（`STABLE_PRODUCT_NAME` = `"Thor Hammer"`）のリンクをクリックする                                                        | 商品詳細ページに遷移する                                                                                                                                                                                                                |
| Act & Assert: カートに追加する                    | "Add to cart" ボタンをクリックする                                                                                                                 | ヘッダーのカート件数バッジが `1` になる（カート追加にトースト通知は無く、このバッジが唯一観測可能な成功シグナル）                                                                                                                       |
| Act & Assert: カートを確認しチェックアウトへ進む  | カートアイコンをクリックし、カート合計が表示されることを確認してから、カートステップの "Proceed to checkout" をクリックする                        | カート合計（`data-test="cart-total"`）が表示される                                                                                                                                                                                      |
| Act & Assert: チェックアウト内でサインインする    | サインインステップのフォームに登録済みアカウントのメール/パスワードを入力してログインし、サインインステップの "Proceed to checkout" をクリックする | "already logged in" メッセージが表示される                                                                                                                                                                                              |
| Act: 国・郵便番号・番地を入力する                 | 住所ステップで国 = `Germany`、郵便番号 = `10115`、番地 = `1` を入力する                                                                            | （street/city/state は手入力しない。次ステップで自動補完を検証する）                                                                                                                                                                    |
| Assert: アプリが street/city/state を自動補完する | 自動補完（デバウンス 300ms + 非同期 `postcode-lookup` API）の完了を待ち、住所ステップの "Proceed to checkout" をクリックする                       | `street="Eckertplatz"`、`city="Heidenheim an der Brenz"`、`state="Rheinland-Pfalz"` が自動入力される（実測値。アプリ側の `FakerPostcodeDriver` が郵便番号を種にした決定的な架空住所を返すため、現実の地理とは無関係）                   |
| Act & Assert: 支払い方法を選択し注文を確定する    | 支払い方法 "Cash on Delivery" を選択し、"Confirm" ボタンを 1 回目クリックする。続けて同じボタンを 2 回目クリックする                               | 1 回目クリック後: "Payment was successful" が表示される（支払い検証のみで注文はまだ作られない）。2 回目クリック後: "Thanks for your order! Your invoice number is" を含む注文確認メッセージが表示される（ここで初めて注文が作成される） |

"Confirm" ボタンは同じハンドラで支払い検証（1 回目）と注文作成（2 回目）という異なる 2 つの処理を担う。注文作成は非冪等な操作（同じ操作を繰り返すと結果が変わる操作）なので、クリックのリトライは行わず 1 回だけに留め、完了表示が出るまでの待機は Web-First Assertions（条件を満たすまで自動でリトライしながら検証する Playwright の機構）に任せる。

### Smoke: トップページ到達性

対象: `tests/smoke.spec.ts`。トップページ（`/`）への `page.goto` が HTTP 200 を返し、`<title>` が空でない文字列を持つことだけを確認する。タイトルの具体的な文言は検証しない。サイト側が文言を変えただけでこのテストが落ちるのを避けるため。
