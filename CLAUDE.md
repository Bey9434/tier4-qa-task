# tier4-qa-task — Claude Code プロジェクト設定

## プロジェクト概要

- https://practicesoftwaretesting.com （E2E テスト練習用の公開サイト）を対象とした E2E テスト自動化プロジェクト
- 内容: 手動テストケースの設計（README 記載）+ Playwright + TypeScript による自動化 + GitHub Actions での CI 実行
- **言語はすべて日本語**（回答・コード内コメント・コミットメッセージ・README）

## コマンド

- `pnpm test` — テスト実行
- ファイル編集後は `.claude/hooks/post-edit-format.sh`（PostToolUse フック）が prettier / eslint --fix を
  自動適用する（prettier / eslint 導入後に有効）。手動での整形実行や整形だけの作業は不要

## アーキテクチャ規約

- **Functional POM**: `pages/` は関数エクスポートのみ。クラス禁止。引数は `Readonly<Page>`
- ロケーター優先順位: `getByRole` > `getByLabel` > `getByPlaceholder` > `getByTestId`（`data-test` 属性）> `getByText`
- 依存方向は `tests/` → `pages/` の一方通行。spec 内での `page.locator()` 直書き禁止
- 情報の書き分け: **コード=How / テストコード=What / コミットログ=Why / コードコメント=Why not**（なぜ当然の選択肢を採らなかったか）。
  コードコメントに How の言い換えや API 名の翻訳を書かない。却下した代替・回避した罠・非自明な制約に絞る。
  ロケーター選定理由・待機戦略・テストデータの根拠は Why not として必ず残す。コメントは self-contained にし、外部メモを読まないと意味が取れない語彙は書かない

## 絶対禁止

- `waitForTimeout`（固定待機）— Web-First Assertions（自動リトライする検証）を使う
- `any` 型
- **商品 ID・商品 URL のハードコード** — このサイトの商品 ID は DB 再シードのたびに変わる ULID。商品名で辿る
- **厳密カウント検証**（例:「商品数が 9 件」）— 公開 admin アカウントで誰でもカタログを変更できるため
- **注文履歴・請求書一覧など蓄積データの検証** — テストアカウントは共有されている。
  検証してよいのは**自分のセッションで作った状態だけ**（例: いま入れたカートの中身、いま完了した注文の確認画面）
- `.env.keys` の内容を出力・ログ・コミットしない

## AI 運用ルール

- 技術的な主張・推奨には**最新の根拠と URL**（公式ドキュメント優先）を付ける。記憶ベースの断言は禁止。
  引用時は **①ページ内のどこからか（節・見出しの特定）②メリット ③既存手段との差分 ④採用すべき理由 ⑤具体的な使い方** の 5 点を書く
- 説明・ドキュメント・コメントは**ジュニアエンジニアが消化できる単位まで分解**する。
  専門用語は薄めず残し、初出時に 1 行定義を添える。「暗黙的にわかるだろう」前提の記述は禁止
- コンテキストを大量に消費する調査や、設計確定後の定型的な実装は
  サブエージェント（`.claude/agents/` に定義）へ委譲し、メインセッションは設計・レビュー・統合に集中する
